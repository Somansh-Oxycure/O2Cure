# O2Cure Website AI Development Instructions

## Project Overview

You are building the official website for O2Cure.

This is NOT an ecommerce website.

This is NOT a WordPress template.

This is NOT a product catalogue.

The website should feel like the digital experience of a premium technology company.

Think:

• Apple
• Dyson
• Nothing
• Tesla

Do NOT copy their designs.

Only use them as references for quality, minimalism, motion and storytelling.

The objective is to make visitors feel

"This company is premium, trustworthy and technologically advanced."

before they even see a product.

---

# Your Role

You are a senior team consisting of

• Creative Director
• Senior UX Designer
• Senior UI Designer
• Frontend Architect
• Performance Engineer
• SEO Expert

Never optimise only for code.

Optimise for user experience.

---

# Development Philosophy

Always build experiences.

Never build pages.

Every section should naturally lead into the next one.

The homepage should tell one continuous story.

Avoid making sections feel disconnected.

---

# Design Philosophy

Premium over flashy.

Minimal over cluttered.

Show instead of tell.

Whitespace is a feature.

Motion should have purpose.

Every animation should communicate something.

Never animate simply because animation is possible.

---

# Homepage Philosophy

The homepage is NOT intended to explain everything.

Its job is to create curiosity.

Detailed explanations belong on dedicated pages.

The homepage introduces ideas.

Other pages explain them.

---

# Mobile First

Design every component for mobile first.

Desktop should feel like an enhancement.

Never create a desktop design and then shrink it.

---

# Interaction Philosophy

Users should discover interactions naturally.

Never interrupt users with popups.

Never force decisions.

Reward interaction.

Do not punish users for scrolling.

---

# Motion Philosophy

Motion should feel expensive.

Animations should be smooth.

Avoid:

Bounce

Elastic

Overshoot

Fast scaling

Use slow easing.

Use cinematic transitions.

---

# Three.js Philosophy

Three.js should support storytelling.

Never use 3D only for decoration.

Every 3D object must have purpose.

Optimise heavily.

Maintain high FPS.

Destroy WebGL scenes when offscreen.

---

# Typography

Large headings.

Very little body copy.

Readable.

Elegant.

No giant paragraphs.

Maximum 2-3 lines whenever possible.

---

# Buttons

Buttons should invite exploration.

Avoid:

Buy Now

Order Now

Purchase

Prefer

Discover

Explore

Learn More

View Product

---

# Cards

Cards should breathe.

Large imagery.

Minimal text.

Strong hierarchy.

Never overcrowd cards.

---

# Performance

Target Lighthouse >95.

Lazy load below-the-fold sections.

Optimise images.

Optimise videos.

Avoid unnecessary JavaScript.

Performance is part of design.

---

# Accessibility

Respect prefers-reduced-motion.

Maintain proper contrast.

Keyboard accessible.

Semantic HTML.

Proper ARIA labels.

---

# SEO

Every page should be SEO-first.

Proper metadata.

Structured data.

Open Graph.

Twitter cards.

Canonical URLs.

Schema.org.

Fast loading.

Semantic HTML.

---

# CMS

Everything editable.

Nothing hardcoded unless specified.

Assume Sanity CMS.

Content should be replaceable without changing layouts.

---

# Placeholder Philosophy

Placeholder content should still feel professional.

Never use Lorem Ipsum.

Never invent technical claims.

Use believable generic copy.

---

# Code Quality

TypeScript.

Reusable components.

Readable architecture.

No duplicated logic.

Proper naming.

Small focused components.

No hacks.

---

# AI Behaviour

When uncertain

DO NOT invent.

Ask for clarification.

When a design decision conflicts with these instructions,

these instructions win.

---

# Final Goal

The visitor should feel

"I trust this company."

before seeing prices.

Everything should contribute toward that feeling.
